# 面向长周期应用开发的编排框架设计

作者：Prithvi Rajasekaran，Anthropic Labs 团队成员。

过去几个月，我一直在研究两个相互关联的问题：让 Claude 产出高质量的前端设计，以及让它在无人干预的情况下独立构建完整应用。这项工作源于我们团队早期在前端设计能力和长周期编码智能体编排框架（harness）上的探索——通过提示工程和编排框架设计，我和同事们将 Claude 的表现提升到了远超基线的水平，但两条路最终都触及了天花板。

为了突破瓶颈，我开始探索新的 AI 工程方法，希望找到能同时适用于两个截然不同领域的通用范式：一个由主观审美定义，另一个由可验证的正确性和可用性主导。受生成对抗网络（GAN）的启发，我设计了一套包含生成器（generator）和评估器（evaluator）的多智能体架构。要让评估器既能稳定打分又具备审美判断力，首先得建立一套评分标准，把"这个设计好不好？"这样的主观判断转化为具体的、可评分的维度。

随后，我把这些技术应用到了长周期自主编码场景中，并沿用了早期编排框架的两条核心经验：将构建任务拆解为可控的小块，以及使用结构化工件（artifact）在会话之间传递上下文。最终成果是一个三智能体架构——规划器（planner）、生成器和评估器——能够在长达数小时的自主编码会话中产出功能丰富的全栈应用。

## 为什么朴素的实现方式行不通

我们此前已经展示过，编排框架设计对长周期智能体编码的效果有着显著影响。在早期的一次实验中，我们用一个初始化智能体将产品规格书拆解为任务列表，再由编码智能体逐个功能实现，并在会话之间传递工件以延续上下文。更广泛的开发者社区也得出了类似的结论，比如"Ralph Wiggum"方法就利用钩子或脚本让智能体保持在持续迭代的循环中。

但有些问题始终挥之不去。面对更复杂的任务，智能体仍然会随着时间推移逐渐跑偏。深入分析后，我们观察到了两种常见的失败模式。

第一种是模型在长任务中随着上下文窗口被填满而逐渐丧失连贯性（可参考我们关于上下文工程（context engineering）的文章）。部分模型还会出现"上下文焦虑（context anxiety）"——当它们认为自己接近上下文窗口上限时，就会提前收工，草草了事。上下文重置（context reset）可以解决这两个问题：彻底清空上下文窗口、启动一个全新的智能体，同时通过结构化交接（structured handoff）将前一个智能体的状态和后续步骤传递过去。

这和上下文压缩（compaction）不同。压缩是把对话早期的部分就地摘要，让同一个智能体在缩短后的历史上继续工作。压缩虽然保留了连续性，却无法给智能体一个干净的起点，这意味着上下文焦虑依然可能存在。重置则提供了一个彻底干净的起点，代价是交接工件必须包含足够的状态信息，让下一个智能体能顺利接手。在早期测试中，我们发现 Claude Sonnet 4.5 的上下文焦虑非常明显，仅靠压缩不足以支撑长任务的高质量表现，因此上下文重置成了编排框架设计中不可或缺的一环。这解决了核心问题，但也给每次运行增加了编排复杂度、token 开销和延迟。

第二个问题是我们之前没有专门讨论过的：自我评估（self-evaluation）。当你让智能体评价自己的产出时，它们往往会信心满满地自夸——哪怕在人类看来质量明显平庸。这个问题在设计这类主观任务中尤为突出，因为不存在像软件测试那样的二元校验手段。一个布局是精致还是平庸，全凭判断，而智能体在给自己打分时总是倾向于往高了打。

不过，即使是在有可验证结果的任务上，智能体在执行过程中也时常表现出糟糕的判断力，拖累整体表现。把"做事的智能体"和"评判的智能体"分开，被证明是解决这个问题的有力杠杆。当然，仅仅分开并不会立刻消除那种"手下留情"的倾向——评估器毕竟还是一个 LLM，天然对 LLM 生成的内容更宽容。但事实证明，调教一个独立的评估器让它变得挑剔，比让生成器学会自我批评要容易得多。而一旦有了外部反馈，生成器就有了具体的改进方向。

## 前端设计：让主观质量变得可评分

我从前端设计开始实验，因为自我评估的问题在这个领域最为明显。在没有任何干预的情况下，Claude 通常会倾向于安全、可预测的布局——技术上能用，视觉上却毫无亮点。

构建前端设计编排框架的过程中，有两个关键洞察。第一，虽然审美不可能完全被量化为一个分数——个人品味永远因人而异——但通过编码设计原则和偏好的评分标准，设计质量是可以被提升的。"这个设计美不美？"很难给出一致的回答，但"这个设计是否遵循了我们的优秀设计原则？"就给了 Claude 一个具体的评判依据。第二，把前端生成和前端评分分开，我们就能构建一个反馈循环（feedback loop），驱动生成器产出越来越好的结果。

基于这些思考，我编写了四条评分标准，同时写入生成器和评估器的提示词中：

**设计质量：** 设计是否呈现为一个有机的整体，而非零部件的拼凑？优秀的设计意味着色彩、排版、布局、图像和其他细节共同营造出独特的氛围和辨识度。

**原创性：** 是否有自主设计决策的痕迹，还是只是模板布局、组件库默认样式和 AI 味（AI slop）的堆砌？一个人类设计师应该能识别出刻意的创意选择。未经修改的默认组件——或者紫色渐变覆盖白色卡片这类 AI 生成的典型特征——在这里直接不及格。

**工艺：** 技术执行力：排版层级、间距一致性、色彩和谐、对比度。这是能力检验而非创意检验。大多数合理的实现默认就能拿到不错的分数；不及格意味着基本功有问题。

**功能性：** 与视觉无关的可用性。用户能否理解界面的用途，找到主要操作入口，无需猜测就完成任务？

我刻意加重了设计质量和原创性的权重，弱化了工艺和功能性。Claude 在工艺和功能性上默认表现就不错，模型天然具备这方面的技术能力。但在设计和原创性上，Claude 的产出往往平淡无奇。评分标准明确惩罚了高度雷同的 AI 味模式，而通过加大设计和原创性的权重，推动模型去承担更多审美风险。

我用包含详细评分拆解的少样本示例来校准评估器。这确保了评估器的判断与我的偏好保持一致，也减少了跨迭代的分数漂移。

整个循环构建在 Claude Agent SDK 之上，编排逻辑因此非常简洁。生成器智能体首先根据用户提示创建一个 HTML/CSS/JS 前端。评估器则接入了 Playwright MCP，可以直接与实时页面交互，然后对每条标准打分并撰写详细的评语。实际运行中，评估器会自主浏览页面，截图并仔细研究实现细节，然后才给出评估。这些反馈会回流给生成器，作为下一轮迭代的输入。每次生成我跑 5 到 15 轮迭代，每一轮通常都会推动生成器朝着更具辨识度的方向演进，因为它在持续回应评估器的批评。由于评估器是在真实浏览页面而非给静态截图打分，每个循环都需要实打实的时间。完整的运行最长可达四个小时。我还指示生成器在每次评估后做出策略性决策：如果分数趋势向好，就打磨当前方向；如果当前路线走不通，就果断转向一个全新的美学风格。

跨多次运行来看，评估器的评分在迭代过程中逐步提升，最终趋于平稳，但仍有上升空间。有些生成是渐进式打磨，有些则在迭代之间发生了急剧的美学转向。

评分标准的措辞对生成器的引导超出了我的预期。比如加入"最好的设计是博物馆级别"这样的表述，会把设计推向某种特定的视觉趋同，说明标准中的提示语言直接塑造了产出的风格特征。

虽然分数总体上随迭代提升，但这个过程并不总是线性的。后期的实现整体往往更好，但我经常发现自己更喜欢中间某一轮的版本，而不是最终版本。实现的复杂度也随着轮次逐渐攀升，生成器在评估器反馈的驱动下不断挑战更有野心的方案。值得注意的是，即使在第一轮迭代中，产出就已经明显优于完全没有提示工程的基线，这说明评分标准和相关的语言本身就已经将模型引离了泛化的默认模式，而评估器的反馈则带来了进一步的精进。

一个令人印象深刻的例子是，我让模型为一座荷兰美术馆创建网站。到第九轮迭代时，它已经为一座虚构博物馆做出了一个简洁的深色主题着陆页。页面视觉上很精致，但基本在我的预期范围内。然后到了第十轮，它彻底推翻了之前的方案，将网站重新构想为一个空间体验：用 CSS 透视渲染出一个棋盘格地板的 3D 房间，画作以自由形式悬挂在墙壁上，用门廊式导航在不同展厅之间穿梭，取代了传统的滚动或点击。这种创意上的飞跃，是我此前在单次生成中从未见过的。

## 扩展到全栈编码

有了这些发现，我把这套 GAN 启发的模式应用到了全栈开发中。生成器-评估器循环天然映射到软件开发生命周期——代码审查和 QA 在结构上扮演着与设计评估器完全相同的角色。

### 架构设计

在早期的长周期编排框架中，我们已经用初始化智能体、逐功能构建的编码智能体以及会话间的上下文重置，解决了多会话编码的连贯性问题。上下文重置是一个关键突破：当时的编排框架使用 Sonnet 4.5，它表现出前面提到的上下文焦虑倾向。设计一个能在上下文重置之间良好运作的编排框架，是保持模型不跑偏的关键。Opus 4.5 在很大程度上自行消除了这种行为，所以在这次的编排框架中我得以完全去掉上下文重置。智能体在整个构建过程中作为一个连续会话运行，上下文增长则由 Claude Agent SDK 的自动压缩机制来处理。

在这项工作中，我在原有编排框架的基础上构建了一个三智能体系统，每个智能体针对我在之前运行中观察到的特定短板。系统包含以下智能体角色：

**规划器：** 早期的长周期编排框架要求用户预先提供详细的规格书。我想把这一步自动化，于是创建了一个规划器智能体，它接收一段 1-4 句的简单描述，展开为完整的产品规格书。我在提示中要求它在范围上保持雄心，同时聚焦于产品上下文和高层技术设计，而非细粒度的技术实现。之所以这样设计，是因为我担心如果规划器试图预先指定过细的技术细节，一旦出错，规格书中的错误就会级联传导到下游实现中。更明智的做法是约束智能体要交付什么，至于怎么走到那里，让它们在工作过程中自己摸索。我还要求规划器寻找机会将 AI 功能融入产品规格书。

**生成器：** 早期编排框架中逐个功能推进的方式在范围管理上效果很好。我在这里采用了类似的模型，指示生成器以冲刺（sprint）为单位工作，每次从规格书中选取一个功能。每个冲刺使用 React、Vite、FastAPI 和 SQLite（后来换成 PostgreSQL）技术栈来实现应用，生成器在每个冲刺结束时被要求自我评估，然后移交给 QA。它还配备了 git 做版本控制。

**评估器：** 早期编排框架产出的应用往往看起来很惊艳，但当你真正上手使用时，还是会有不少 bug。为了捕获这些问题，评估器使用 Playwright MCP 像真实用户一样点击浏览运行中的应用，测试 UI 功能、API 端点和数据库状态。然后它根据发现的 bug 和一组改编自前端实验的评分标准——涵盖产品深度、功能性、视觉设计和代码质量——对每个冲刺进行评分。每条标准都有一个硬性阈值，任何一条低于阈值，该冲刺就判定失败，生成器会收到关于问题所在的详细反馈。

在每个冲刺开始前，生成器和评估器会协商一份冲刺契约（sprint contract）：在写任何代码之前，先就这块工作的"完成"标准达成一致。之所以需要这一步，是因为产品规格书有意保持在高层级，而我需要一个环节来弥合用户故事和可测试实现之间的鸿沟。生成器提出它打算构建什么以及如何验证成功，评估器审核这个提案以确保生成器在做正确的事情。双方反复迭代直至达成共识。

沟通通过文件进行：一个智能体写入文件，另一个智能体读取后在同一文件中回应，或者创建新文件供上一个智能体读取。生成器随后按照达成一致的契约进行构建，完成后移交给 QA。这种方式既忠实于规格书，又避免了过早指定实现细节。

### 运行编排框架

这个编排框架的第一版，我用的是 Claude Opus 4.5，把同样的用户提示词分别丢给完整编排框架和单智能体系统跑了一遍做对比。选 Opus 4.5 是因为实验启动时，它是我们手头最强的编码模型。

我写了下面这段提示词来生成一个复古电子游戏制作器：

> Create a 2D retro game maker with features including a level editor, sprite editor, entity behaviors, and a playable test mode.

下表展示了编排框架类型、运行时长和总成本。

| 编排框架 | 时长 | 成本 |
|---|---|---|
| 单智能体 | 20 分钟 | $9 |
| 完整编排框架 | 6 小时 | $200 |

编排框架的成本是单智能体的 20 倍以上，但输出质量的差距一眼就能看出来。

我期望得到的是这样一个界面：能构建关卡及其组成部分（精灵、实体、地块布局），然后按下播放键就能实际游玩关卡。我先打开了单智能体的产出，初始界面看着确实符合预期。

然而，当我逐个功能点击过去，问题就开始冒头了。布局浪费了大量空间——固定高度的面板让视口大部分区域空空荡荡。工作流程也很死板：当我试图在关卡中放置内容时，系统提示需要先创建精灵和实体，但界面上没有任何引导告诉你该按这个顺序来。更要命的是，游戏本身是坏的。我的实体确实出现在了屏幕上，但完全不响应输入。翻了代码才发现，实体定义和游戏运行时之间的接线是断的，而且界面上没有任何迹象提示问题出在哪里。

评估完单智能体的结果后，我转向了编排框架的产出。这次运行同样从那一句话的提示词起步，但规划器将它扩展成了一份包含 16 个功能、横跨十个冲刺的完整规格说明——远远超出了单智能体所尝试的范围。除了核心的编辑器和游玩模式，规格说明还包括精灵动画系统、行为模板、音效和音乐、AI 辅助的精灵生成器和关卡设计器，以及带可分享链接的游戏导出功能。我让规划器读取了我们的前端设计技能文档，它据此为整个应用创建了一套视觉设计语言并纳入规格。在每个冲刺中，生成器和评估器会协商一份冲刺契约，定义该冲刺的具体实现细节以及用于验证完成度的可测试行为。

应用立刻就展现出了比单智能体更高的完成度和流畅度。画布占满了整个视口，面板大小合理，界面有一套一致的视觉风格，紧跟规格中的设计方向。我在单智能体版本中看到的一些笨拙之处仍然存在——工作流程依然没有明确告知用户应该先建好精灵和实体再去填充关卡，我还是得自己摸索着来。这更像是基础模型在产品直觉上的短板，而不是编排框架要解决的问题。不过它也指出了一个方向：在编排框架内做有针对性的迭代，有望进一步提升输出质量。

逐个体验编辑器的过程中，新版本的优势变得更加明显。精灵编辑器更加丰富和完善——工具面板更整洁、颜色选择器更好用、缩放控件也更顺手。

因为我要求规划器在规格中织入 AI 功能，应用还自带了 Claude 集成，让我可以通过提示词生成游戏的各个组件。这大大加快了整个工作流程。

最大的差别在于游玩模式。我真的能移动自己的角色并游玩了。物理效果还有些粗糙——我的角色跳上平台后跟平台重叠了，直觉上就觉得不对——但核心玩法是跑通的，而单智能体版本完全没做到。移动了一会儿后，我也碰到了 AI 关卡构建的一些局限：有一堵大墙怎么也跳不过去，于是卡住了。这说明还有一些常识性的改进和边界情况有待处理，编排框架可以继续在这些细节上打磨。

翻阅日志后可以清楚地看到，评估器让实现始终与规格保持一致。每个冲刺中，它都会逐条检查冲刺契约里的测试标准，并通过 Playwright 操作运行中的应用，对任何偏离预期行为的地方提交 bug。契约非常细致——仅冲刺 3 就有 27 条标准覆盖关卡编辑器——而评估器的发现也足够具体，不需要额外调查就能直接着手修复。下表展示了评估器识别出的几个问题示例：

| 契约标准 | 评估器发现 |
|---|---|
| Rectangle fill tool allows click-drag to fill a rectangular area with selected tile | FAIL — Tool only places tiles at drag start/end points instead of filling the region. fillRectangle function exists but isn't triggered properly on mouseUp. |
| User can select and delete placed entity spawn points | FAIL — Delete key handler at LevelEditor.tsx:892 requires both selection and selectedEntityId to be set, but clicking an entity only sets selectedEntityId. Condition should be selection \|\| (selectedEntityId && activeLayer === 'entity'). |
| User can reorder animation frames via API | FAIL — PUT /frames/reorder route defined after /{frame_id} routes. FastAPI matches 'reorder' as a frame_id integer and returns 422: "unable to parse string as an integer." |

让评估器达到这个水平，花了相当多的功夫。开箱即用的 Claude 并不是一个好的 QA 智能体。在早期运行中，我眼睁睁看着它发现了真实的问题，然后说服自己"其实也没那么严重"，照样放行。它的测试也倾向于浮在表面，不会深入探测边界情况，更隐蔽的 bug 就这么溜过去了。调优循环是这样的：读评估器的日志，找到它的判断与我的判断产生分歧的地方，然后更新 QA 的提示词来修正这些偏差。这样的开发循环跑了好几轮，评估器的评分才达到我认为合理的水平。即便如此，编排框架的输出仍然暴露了模型 QA 能力的天花板：零星的布局瑕疵、某些地方不够直观的交互，以及嵌套较深的功能中评估器未充分测试到的遗留 bug。在进一步调优上，显然还有不少验证空间可以挖掘。但对比单智能体——连应用的核心功能都跑不通——提升显而易见。

## 编排框架的迭代

第一组结果令人鼓舞，但编排框架本身也够臃肿、够慢、够贵的。合乎逻辑的下一步是在不牺牲性能的前提下做减法。这一方面是常识，另一方面源于一个更普遍的原则：编排框架中的每个组件都编码了一个关于"模型自身做不到什么"的假设，而这些假设值得反复检验——既因为它们可能本来就不成立，也因为随着模型的进步，它们会迅速过时。我们的博客文章《Building Effective Agents》将底层理念概括为"找到尽可能简单的方案，只在需要时才增加复杂度"——对任何维护智能体编排框架的人来说，这是一个反复应验的模式。

在第一次简化尝试中，我大刀阔斧地砍掉了编排框架的大部分组件，还试了几个有创意的新想法，但没能复现原版的性能水平。更棘手的是，很难判断编排框架中哪些部分才是真正承重的，又是以什么方式在起作用。基于这次教训，我转向了更系统的路子：每次只移除一个组件，然后观察它对最终结果的影响。

就在我进行这些迭代循环的同时，我们发布了 Opus 4.6，这为进一步降低编排框架复杂度提供了额外动力。有充分理由相信 4.6 需要的脚手架会比 4.5 更少。引用我们的发布博客："[Opus 4.6] 规划更审慎，能更持久地执行智能体任务，在大型代码库中运行更可靠，并且具备更强的代码审查和调试能力来发现自身的错误。"它在长上下文检索方面也有了显著提升。而这些，恰恰都是编排框架此前一直在补充的能力。

### 移除冲刺结构

我从移除冲刺结构开始。冲刺结构此前的作用是将工作分解为模型能够连贯处理的小块。鉴于 Opus 4.6 的进步，有充分理由相信模型能原生地完成这些工作，不再需要这种拆解。

我保留了规划器和评估器，因为它们各自仍有明显的价值。没有规划器的话，生成器会严重缩水：面对原始提示词，它会直接开工而不先制定规格，最终做出的应用远不如有规划器时那么丰富。

移除冲刺结构后，我将评估器改为运行结束时做一次性评审，而不再按冲刺逐一评分。由于模型能力大幅提升，评估器的承重程度也随之发生了变化——它的实用性取决于任务在模型自主能力边界中所处的位置。在 4.5 上，这个边界离得很近：我们的构建任务已经处于生成器单独能做好的极限，评估器在整个构建过程中都捕获了有意义的问题。到了 4.6，模型的原始能力上了一个台阶，边界向外推移。那些过去需要评估器兜底才能连贯实现的任务，现在往往在生成器自己的能力范围之内，评估器对它们而言成了不必要的开销。但对于仍然处在生成器能力边缘的部分，评估器依然带来了实实在在的提升。

实际启示是：评估器并非一个固定的"要或不要"的决策。当任务超出当前模型能可靠独立完成的范围时，它才值得投入。

在结构简化的同时，我还增加了提示词来改进编排框架构建 AI 功能的方式——具体来说，让生成器构建一个真正的智能体，能够通过工具驱动应用自身的功能。这花了相当多的迭代，因为相关知识足够新，Claude 的训练数据对此覆盖有限。但经过充分调优，生成器确实能正确地构建智能体了。

### 更新后的编排框架结果

为了检验更新后的编排框架，我用以下提示词生成了一个数字音频工作站（DAW）——一个用于作曲、录音和混音的音乐制作程序：

> Build a fully featured DAW in the browser using the Web Audio API.

这次运行仍然耗时且昂贵——大约 4 小时，token 成本 $124。

大部分时间花在了构建阶段。生成器连贯运行了超过两个小时，不再需要 Opus 4.5 时代那种冲刺拆解。

| 智能体与阶段 | 时长 | 成本 |
|---|---|---|
| 规划器 | 4.7 分钟 | $0.46 |
| 构建（第 1 轮） | 2 小时 7 分钟 | $71.08 |
| QA（第 1 轮） | 8.8 分钟 | $3.24 |
| 构建（第 2 轮） | 1 小时 2 分钟 | $36.89 |
| QA（第 2 轮） | 6.8 分钟 | $3.09 |
| 构建（第 3 轮） | 10.9 分钟 | $5.88 |
| QA（第 3 轮） | 9.6 分钟 | $4.06 |
| **V2 编排框架总计** | **3 小时 50 分钟** | **$124.70** |

和前一版一样，规划器将一行提示词扩展成了完整的规格说明。从日志中可以看到，生成器在应用规划和智能体设计、接线，以及交付 QA 前的自测等方面都做得不错。

话虽如此，QA 智能体仍然捕获了真实的缺漏。在第一轮反馈中，它指出：

> This is a strong app with excellent design fidelity, solid AI agent, and good backend. The main failure point is Feature Completeness — while the app looks impressive and the AI integration works well, several core DAW features are display-only without interactive depth: clips can't be dragged/moved on the timeline, there are no instrument UI panels (synth knobs, drum pads), and no visual effect editors (EQ curves, compressor meters). These aren't edge cases — they're the core interactions that make a DAW usable, and the spec explicitly calls for them.

第二轮反馈中，它再次揪出了几处功能缺口：

> Remaining gaps:
> - Audio recording is still stub-only (button toggles but no mic capture)
> - Clip resize by edge drag and clip split not implemented
> - Effect visualizations are numeric sliders, not graphical (no EQ curve)

放手让生成器自己干的时候，它仍然容易遗漏细节或把功能做成存根（stub）。QA 在捕获这些最后一公里的问题上依然在发挥价值，让生成器有机会回头修补。

按照提示词的描述，我期望的是一个能创建旋律、和声和鼓点节奏的程序，可以将它们编排成一首歌，并在过程中随时得到内置智能体的帮助。

这个应用离专业的音乐制作程序还差得很远，智能体的作曲能力显然还需要大量打磨。而且 Claude 实际上听不见声音，这让 QA 反馈循环在音乐品味这件事上力不从心。

但最终的应用具备了一个可用的音乐制作程序的所有核心要素：一个能在浏览器中运行的编曲视图、混音器和走带控制。不止于此——我还完全通过提示词拼出了一小段歌曲：智能体设置了速度和调式，铺下旋律，构建了鼓轨，调整了混音器电平，加上了混响。歌曲创作的核心原语都已就位，智能体能够自主驱动它们，用工具从头到尾完成一段简单的制作。

## 未来展望

随着模型不断进步，我们大致可以预期它们能胜任更长时间、更复杂的任务。在某些场景下，这意味着模型外围的脚手架会随时间推移而变得不那么重要——开发者可以坐等下一代模型，看着某些问题自行消解。但硬币的另一面是，模型越强大，编排框架能施展的空间就越大，足以完成模型在基线水平上无法独立搞定的复杂任务。

带着这个认知，这项工作中有几条经验值得带走。首先，针对你所使用的模型做实验、在真实问题上通读它的执行轨迹、调优它的表现以达到你想要的效果——这始终是好的实践。其次，面对更复杂的任务时，将其拆解并为各个环节指派专门的智能体，有时能打开新的提升空间。最后，当新模型落地时，重新审视你的编排框架通常是明智之举：拆掉那些对性能不再承重的部件，同时装上新的组件，去实现此前不可能达到的更强能力。

从这项工作中，我得出了一个坚定的判断：随着模型进步，有趣的编排框架组合空间并不会缩小——它在迁移。对 AI 工程师而言，真正有趣的工作，是持续发现下一个新颖的组合。
