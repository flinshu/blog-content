# 面向长周期应用开发的编排框架设计

作者：Prithvi Rajasekaran，Anthropic Labs 团队成员。

过去几个月，我一直在研究两个相互关联的问题：让 Claude 产出高质量的前端设计，以及让它在无人干预的情况下构建完整的应用程序。这项工作源于我们早期在前端设计技能和长周期编码智能体编排框架（harness）上的探索。在那些尝试中，我和同事们通过提示工程和编排框架设计，将 Claude 的表现大幅提升到了基线之上——但最终都触及了天花板。

为了突破这些瓶颈，我开始寻找能够跨越两个截然不同领域的 AI 工程新方法：一个由主观审美定义，另一个由可验证的正确性和可用性定义。受生成对抗网络（GAN）的启发，我设计了一套包含生成器（generator）和评估器（evaluator）的多智能体架构。要构建一个能够可靠评分——且具备审美品味的评估器，首先需要制定一套评判标准，将"这个设计好不好？"这样的主观判断转化为具体的、可评分的维度。

随后，我将这些技术应用到了长周期自主编码任务中，并沿用了早期编排框架工作的两条经验：将构建过程拆解为可控的小块，以及使用结构化工件（artifact）在会话之间传递上下文。最终的成果是一套三智能体架构——规划器（planner）、生成器（generator）和评估器（evaluator）——它能够在长达数小时的自主编码会话中产出功能丰富的全栈应用。

## 为什么简单的实现方式行不通

我们此前已经证明，编排框架设计对长周期智能体编码的效果有显著影响。在早期的一次实验中，我们使用一个初始化智能体将产品规格拆解为任务列表，再由编码智能体逐个实现功能，并通过工件在会话间传递上下文。更广泛的开发者社区也得出了类似的洞察——例如"Ralph Wiggum"方法，通过钩子或脚本让智能体保持在持续迭代的循环中。

但有些问题始终挥之不去。对于更复杂的任务，智能体仍然会随着时间推移逐渐跑偏。在对这个问题进行拆解分析后，我们观察到智能体在执行此类任务时存在两种常见的失败模式。

第一种是模型在处理冗长任务时，随着上下文窗口被填满，会逐渐丧失连贯性（参见我们关于上下文工程（Context Engineering）的文章）。某些模型还会表现出"上下文焦虑"（context anxiety）——当它们认为自己接近上下文长度限制时，会过早地草草收尾。上下文重置（context reset）——即完全清空上下文窗口并启动一个新的智能体，同时通过结构化交接（structured handoff）将前一个智能体的状态和后续步骤传递过去——能够同时解决这两个问题。

这与上下文压缩（compaction）不同。上下文压缩是将对话早期部分就地摘要，让同一个智能体在缩短的历史上继续工作。虽然上下文压缩保留了连续性，但它无法给智能体一个干净的起点，这意味着上下文焦虑仍可能持续存在。上下文重置提供了一个干净的起点，代价是交接工件必须包含足够的状态信息，让下一个智能体能够顺利接手。在我们早期的测试中，我们发现 Claude Sonnet 4.5 的上下文焦虑非常明显，仅靠上下文压缩不足以支撑长任务的良好表现，因此上下文重置成为了编排框架设计中不可或缺的一环。这解决了核心问题，但也给每次编排运行增加了额外的协调复杂度、token 开销和延迟。

第二个问题是我们此前未曾解决的：自我评估（self-evaluation）。当被要求评价自己产出的工作时，智能体倾向于自信满满地夸赞自己的成果——即使在人类观察者看来，质量明显平庸。这个问题在设计等主观任务中尤为突出，因为这类任务不存在像可验证软件测试那样的二元判定标准。一个布局究竟是精致还是平庸，本质上是一种判断，而智能体在给自己打分时总是系统性地偏高。

然而，即使在有可验证结果的任务上，智能体在完成工作过程中有时也会表现出影响其性能的糟糕判断。将执行工作的智能体与评判工作的智能体分离开来，被证明是解决这一问题的有力杠杆。这种分离本身并不会立即消除评分偏宽的倾向——评估器仍然是一个 LLM，天然倾向于对 LLM 生成的输出宽容以待。但事实证明，调教一个独立的评估器使其保持怀疑态度，比让生成器对自己的工作持批判态度要容易得多。而一旦有了外部反馈，生成器就有了具体的改进方向。

## 前端设计：让主观质量变得可评分

我从前端设计开始实验，因为自我评估的问题在这里最为明显。在没有任何干预的情况下，Claude 通常会倾向于安全、可预测的布局——技术上可用，但视觉上毫无亮点。

两条洞察塑造了我为前端设计构建的编排框架。第一，虽然审美无法完全被量化为分数——个人品味也总会因人而异——但通过编码了设计原则和偏好的评分标准，审美是可以被改善的。"这个设计美不美？"很难得到一致的回答，但"这个设计是否遵循了我们关于好设计的原则？"就给了 Claude 一个具体的评判依据。第二，通过将前端生成与前端评分分离，我们可以创建一个反馈循环（feedback loop），推动生成器产出更强的结果。

基于这些思路，我编写了四条评分标准，同时提供给生成器和评估器智能体作为提示词的一部分：

**设计质量：** 设计是否给人一种浑然一体的感觉，而非零散部件的拼凑？这一项得分高，意味着色彩、排版、布局、图像及其他细节组合在一起，创造出了独特的氛围和辨识度。

**原创性：** 是否有自主设计决策的痕迹，还是只是模板布局、组件库默认值和 AI 生成的套路？一个人类设计师应该能识别出刻意的创意选择。未经修改的原装组件——或者 AI 生成的典型特征，比如白色卡片上的紫色渐变——在这一项上不及格。

**工艺：** 技术执行层面：排版层级、间距一致性、色彩和谐度、对比度。这是能力检验而非创意检验。大多数合理的实现默认就能做到；不及格意味着基本功有问题。

**功能性：** 独立于美学的可用性。用户能否理解界面的功能、找到主要操作入口、不用猜测就能完成任务？

我刻意将设计质量和原创性的权重置于工艺和功能性之上。Claude 在工艺和功能性方面默认就能拿到不错的分数，因为所需的技术能力本就是模型的强项。但在设计和原创性方面，Claude 的产出往往乏善可陈。这些评分标准明确惩罚了高度同质化的"AI 味"（AI slop）模式，并通过加大设计和原创性的权重，推动模型去承担更多审美上的风险。

我使用附带详细评分拆解的少样本示例来校准评估器。这确保了评估器的判断与我的偏好保持一致，并减少了跨迭代的评分漂移。

我基于 Claude Agent SDK 构建了这个循环，这让编排逻辑保持简洁。生成器智能体首先根据用户提示词创建一个 HTML/CSS/JS 前端。我为评估器配备了 Playwright MCP，使其能够在评分和撰写详细评语之前，直接与实时页面进行交互。实际运行中，评估器会自行浏览页面、截图并仔细审视实现效果，然后再给出评估。这些反馈作为下一次迭代的输入回传给生成器。每次生成我运行 5 到 15 轮迭代，每轮迭代通常都会推动生成器朝着更具辨识度的方向演进，因为它在回应评估器的批评。由于评估器是在主动浏览页面而非对着静态截图评分，每个周期都需要实际的时间。完整的运行最长可达四小时。我还指示生成器在每次评估后做出一个策略性决策：如果分数趋势向好就继续优化当前方向，如果当前路线行不通就彻底转向一种全新的美学风格。

跨多次运行来看，评估器的评分在迭代过程中呈上升趋势，随后进入平台期，但仍有提升空间。有些生成过程是渐进式的优化，有些则在迭代之间发生了剧烈的美学转向。

评分标准的措辞对生成器的引导作用超出了我的预期。加入"最好的设计应该达到博物馆级别"这样的表述后，设计开始朝着某种特定的视觉风格趋同。这说明与评分标准相关的提示词直接塑造了产出的特质。

虽然分数总体上随迭代改善，但这一趋势并不总是线性的。后期的实现整体上往往更好，但我经常遇到中间某一轮的迭代比最后一轮更合我心意的情况。实现的复杂度也倾向于在轮次间递增，因为生成器在评估器反馈的驱动下会尝试更有野心的方案。即使在第一轮迭代，产出也明显优于完全没有提示词引导的基线水平。这说明评分标准本身及其相关的表述，在评估器反馈带来进一步优化之前，就已经将模型从通用默认模式中拉了出来。

在一个值得一提的案例中，我让模型为一家荷兰艺术博物馆创建网站。到第九轮迭代时，它产出了一个为虚构博物馆设计的简洁暗色风格落地页。页面视觉上很精致，但基本在我的预期范围内。然而在第十轮，它彻底推翻了之前的方案，将网站重新构想为一种空间体验：一个用 CSS 透视渲染的带方格地板的 3D 房间，画作以自由排列的方式挂在墙上，房间之间通过门廊导航而非滚动或点击。这是我从未在单次生成中见过的那种创造性飞跃。

## 扩展到全栈编码

有了这些发现，我将这种受 GAN 启发的模式应用到了全栈开发中。生成器-评估器循环天然映射到软件开发生命周期——其中代码评审和 QA 扮演着与设计评估器相同的结构性角色。

### 架构

在我们早期的长周期编排框架中，我们通过初始化智能体、逐功能工作的编码智能体以及会话间的上下文重置，解决了多会话编码的连贯性问题。上下文重置是一个关键突破：当时的编排框架使用的是 Sonnet 4.5，它明显表现出前文提到的"上下文焦虑"倾向。设计一个能够跨上下文重置良好运作的编排框架，是让模型保持专注的关键。Opus 4.5 在很大程度上自行消除了这一行为，因此我得以从这个编排框架中完全移除上下文重置。智能体在整个构建过程中作为一个连续会话运行，上下文增长则由 Claude Agent SDK 的自动压缩机制来处理。

在这项工作中，我以原始编排框架为基础，构建了一个三智能体系统，每个智能体针对我在先前运行中观察到的特定短板。系统包含以下智能体角色：

**规划器：** 我们之前的长周期编排框架要求用户预先提供详细的规格说明。我想自动化这一步骤，于是创建了一个规划器智能体，它接收一段 1-4 句话的简单提示，将其扩展为完整的产品规格。我在提示中要求它在范围上保持雄心勃勃，同时聚焦于产品上下文和高层技术设计，而非细粒度的技术实现。这样做是因为我担心：如果规划器试图在前期指定过于详细的技术细节，一旦出错，规格中的错误就会级联传导到下游实现中。更明智的做法似乎是约束智能体要交付的成果，让它们在工作中自行摸索路径。我还要求规划器在产品规格中寻找织入 AI 功能的机会。

**生成器：** 早期编排框架中逐功能推进的方式在范围管理上效果不错。我在这里采用了类似的模式，指示生成器以冲刺（sprint）的方式工作，每次从规格中选取一个功能来实现。每个冲刺使用 React、Vite、FastAPI 和 SQLite（后来改为 PostgreSQL）技术栈来构建应用，生成器在每个冲刺结束时被要求自我评估工作成果，然后交接给 QA。它还使用 git 进行版本控制。

**评估器：** 早期编排框架产出的应用往往看起来很惊艳，但当你真正上手使用时，仍然存在实际的 bug。为了捕获这些问题，评估器使用 Playwright MCP 像真实用户一样点击浏览运行中的应用，测试 UI 功能、API 端点和数据库状态。然后，它根据发现的 bug 和一套改编自前端实验的评分标准，对每个冲刺进行评分。这套标准在这里覆盖了产品深度、功能性、视觉设计和代码质量。每项标准都有一个硬性阈值，任何一项低于阈值，该冲刺就算失败，生成器会收到关于具体哪里出了问题的详细反馈。

在每个冲刺开始之前，生成器和评估器会协商一份冲刺契约（sprint contract）：在编写任何代码之前，就"完成"对这块工作意味着什么达成一致。之所以设立这一步骤，是因为产品规格是有意保持高层概括的，而我需要一个环节来弥合用户故事与可测试实现之间的鸿沟。生成器提出它打算构建什么以及如何验证成功，评估器审查该提案以确保生成器在构建正确的东西。双方反复迭代直至达成一致。

沟通通过文件进行：一个智能体写入文件，另一个智能体读取文件并在该文件中回复，或者创建一个新文件供前一个智能体读取。然后，生成器按照达成一致的契约进行开发，完成后将工作交接给 QA。这种方式让工作忠实于规格说明，同时避免了过早地过度指定实现细节。

### 运行编排框架

在这个编排框架的第一版中，我使用了 Claude Opus 4.5，将用户提示词分别在完整编排框架和单智能体系统上运行以进行对比。之所以选择 Opus 4.5，是因为在我开始这些实验时，它是我们最好的编码模型。

我编写了以下提示词来生成一个复古电子游戏制作器：

> Create a 2D retro game maker with features including a level editor, sprite editor, entity behaviors, and a playable test mode.

下表展示了编排框架类型、运行时长和总成本。

| 编排框架 | 时长 | 成本 |
|---|---|---|
| 单智能体 | 20 分钟 | $9 |
| 完整编排框架 | 6 小时 | $200 |

完整编排框架的成本是单智能体的 20 倍以上，但输出质量的差异一目了然。

我期望的是一个能让我构建关卡及其组成部分（精灵、实体、地块布局）的界面，然后按下播放键就能实际游玩关卡。我先打开了单智能体运行的输出，初始应用看起来确实符合这些预期。

然而，当我逐一点击各个功能时，问题开始浮现。布局浪费了大量空间——固定高度的面板让视口大部分区域空空荡荡。工作流程也很僵硬。当我试图在关卡中放置内容时，系统提示我需要先创建精灵和实体，但界面上没有任何引导告诉我应该按照这个顺序操作。更关键的是，实际游戏是坏的。我的实体出现在了屏幕上，但完全不响应输入。深入代码后发现，实体定义和游戏运行时之间的连接是断的，而且界面上没有任何迹象表明问题出在哪里。

评估完单智能体的运行结果后，我把注意力转向了编排框架的运行结果。这次运行同样从那一句话的提示词开始，但规划器步骤将该提示词扩展成了一份包含 16 个功能、分布在十个冲刺中的完整规格说明。它远远超出了单智能体所尝试的范围。除了核心的编辑器和游玩模式之外，规格说明还包括精灵动画系统、行为模板、音效和音乐、AI 辅助的精灵生成器和关卡设计器，以及带可分享链接的游戏导出功能。我让规划器读取了我们的前端设计技能文档，它据此为整个应用创建了一套视觉设计语言，并纳入规格说明。在每个冲刺中，生成器和评估器会协商一份冲刺契约，定义该冲刺的具体实现细节以及用于验证完成度的可测试行为。

这个应用立刻展现出比单智能体更高的完成度和流畅度。画布占满了整个视口，面板尺寸合理，界面有一套一致的视觉风格，与规格说明中的设计方向保持一致。我在单智能体中看到的一些笨拙之处仍然存在——工作流程依然没有清楚地告知用户应该先构建精灵和实体再去填充关卡，我不得不自己摸索。这更像是基础模型在产品直觉上的短板，而非编排框架设计上要解决的问题。不过，这也指出了一个方向：在编排框架内进行有针对性的迭代，有可能进一步提升输出质量。

在体验各个编辑器的过程中，新版本相比单智能体的优势变得更加明显。精灵编辑器更加丰富和完善——工具面板更整洁、颜色选择器更好用、缩放控件也更顺手。

因为我要求规划器在规格说明中融入 AI 功能，应用还内置了 Claude 集成，让我可以通过提示词生成游戏的各个部分。这大大加快了工作流程。

最大的区别在于游玩模式。我真的能够移动自己的实体并游玩游戏了。物理效果还有些粗糙——我的角色跳上平台后和平台重叠了，直觉上感觉不对——但核心功能是跑通了的，而单智能体版本完全没做到这一点。移动了一会儿之后，我确实遇到了 AI 关卡构建的一些局限。有一堵大墙我怎么也跳不过去，于是卡住了。这表明还有一些常识性的改进和边界情况需要处理，编排框架可以进一步打磨这些细节。

阅读日志后可以清楚地看到，评估器让实现始终与规格说明保持一致。每个冲刺中，它都会逐一检查冲刺契约中的测试标准，并通过 Playwright 对运行中的应用进行测试，对任何偏离预期行为的地方提交 bug。这些契约非常细致——仅冲刺 3 就有 27 条标准覆盖关卡编辑器——而评估器的发现也足够具体，无需额外调查就能直接定位和修复。下表展示了评估器识别出的几个问题示例：

| 契约标准 | 评估器发现 |
|---|---|
| Rectangle fill tool allows click-drag to fill a rectangular area with selected tile | FAIL — Tool only places tiles at drag start/end points instead of filling the region. fillRectangle function exists but isn't triggered properly on mouseUp. |
| User can select and delete placed entity spawn points | FAIL — Delete key handler at LevelEditor.tsx:892 requires both selection and selectedEntityId to be set, but clicking an entity only sets selectedEntityId. Condition should be selection \|\| (selectedEntityId && activeLayer === 'entity'). |
| User can reorder animation frames via API | FAIL — PUT /frames/reorder route defined after /{frame_id} routes. FastAPI matches 'reorder' as a frame_id integer and returns 422: "unable to parse string as an integer." |

让评估器达到这个水平花了不少功夫。开箱即用的 Claude 并不是一个好的 QA 智能体。在早期的运行中，我看到它明明识别出了真实的问题，却说服自己这些问题不严重，然后照样通过了审查。它的测试也往往流于表面，不会深入探测边界情况，因此更隐蔽的 bug 经常被放过。调优的循环是这样的：阅读评估器的日志，找到它的判断与我的判断不一致的地方，然后更新 QA 的提示词来解决这些问题。经过好几轮这样的开发循环，评估器的评分才达到我认为合理的水平。即便如此，编排框架的输出仍然暴露了模型 QA 能力的局限：小的布局问题、某些地方不太直观的交互，以及嵌套较深的功能中评估器未充分测试到的隐藏 bug。在进一步调优上显然还有更多的验证空间可以挖掘。但相比于单智能体——连应用的核心功能都跑不通——提升显而易见。

## 编排框架的迭代

第一组编排框架的结果令人鼓舞，但它也过于臃肿、缓慢且昂贵。合乎逻辑的下一步是在不降低性能的前提下寻找简化编排框架的方法。这一方面是常识，另一方面则源于一个更普遍的原则：编排框架中的每个组件都编码了一个关于"模型自身无法做到什么"的假设，而这些假设值得反复检验——既因为它们可能本身就不正确，也因为随着模型的进步，它们会迅速过时。我们的博客文章《Building Effective Agents》提出了一个底层理念："找到尽可能简单的方案，只在需要时才增加复杂度。"对于任何维护智能体编排框架的人来说，这是一个反复出现的模式。

在第一次简化尝试中，我大刀阔斧地削减了编排框架，并尝试了一些有创意的新想法，但未能复现原版的性能。而且，也变得难以判断编排框架设计中哪些部分才是真正承重的（load-bearing），以及它们以何种方式发挥作用。基于这次经验，我转向了更系统的方法：每次只移除一个组件，然后审查它对最终结果的影响。

在我进行这些迭代循环的同时，我们还发布了 Opus 4.6，这为进一步降低编排框架复杂度提供了额外的动力。有充分的理由相信 4.6 所需的脚手架（scaffolding）会比 4.5 更少。引用我们的发布博客："[Opus 4.6] 规划更审慎，能更持久地执行智能体任务，在大型代码库中运行更可靠，并且具备更好的代码审查和调试能力来发现自身的错误。"它在长上下文检索方面也有了显著提升。而这些能力，正是编排框架此前一直在补充的。

### 移除冲刺结构

我首先将冲刺结构整个移除。冲刺结构此前的作用是将工作分解为模型能够连贯处理的单元。鉴于 Opus 4.6 的改进，有充分理由相信模型可以原生地完成这项工作，而无需这种分解。

我保留了规划器和评估器，因为它们各自仍然有明显的价值。没有规划器的话，生成器的规划范围会不足：面对原始提示词，它会直接开始构建而不先制定规格说明，最终做出的应用功能远不如规划器版本丰富。

移除冲刺结构后，我将评估器改为在运行结束时做一次性评审，而非按冲刺逐一评分。由于模型能力大幅提升，评估器在不同运行中的承重程度也发生了变化——它的实用性取决于任务在模型自主可靠完成的能力边界中所处的位置。在 4.5 上，这个边界很近：我们的构建已经处于生成器单独能做好的极限，评估器在整个构建过程中捕获了有意义的问题。在 4.6 上，模型的原始能力提升了，边界向外推移。那些过去需要评估器检查才能保证连贯实现的任务，现在往往在生成器自己的能力范围之内，评估器对这些任务变成了不必要的开销。但对于仍处在生成器能力边缘的部分，评估器依然带来了实实在在的提升。

实际启示是：评估器并非一个固定的"要或不要"的决策。当任务超出当前模型能可靠独立完成的范围时，它就值得投入成本。

在结构简化的同时，我还增加了提示词优化，改进编排框架将 AI 功能集成到每个应用中的方式——具体来说，是让生成器构建一个真正的智能体，能够通过工具驱动应用自身的功能。这花了大量迭代才实现，因为相关知识足够新，Claude 的训练数据覆盖有限。但经过充分调优后，生成器确实能正确地构建智能体了。

### 更新后的编排框架结果

为了测试更新后的编排框架，我使用了以下提示词来生成一个数字音频工作站（DAW），即一个用于作曲、录音和混音的音乐制作程序：

> Build a fully featured DAW in the browser using the Web Audio API.

这次运行仍然耗时且昂贵，大约 4 小时，token 成本为 $124。

大部分时间花在了构建器上，它连贯运行了超过两个小时，不再需要 Opus 4.5 所依赖的冲刺分解。

| 智能体与阶段 | 时长 | 成本 |
|---|---|---|
| 规划器 | 4.7 分钟 | $0.46 |
| 构建（第 1 轮） | 2 小时 7 分钟 | $71.08 |
| QA（第 1 轮） | 8.8 分钟 | $3.24 |
| 构建（第 2 轮） | 1 小时 2 分钟 | $36.89 |
| QA（第 2 轮） | 6.8 分钟 | $3.09 |
| 构建（第 3 轮） | 10.9 分钟 | $5.88 |
| QA（第 3 轮） | 9.6 分钟 | $4.06 |
| **V2 编排框架总计** | **3 小时 50 分钟** | **$124.70** |

和前一版编排框架一样，规划器将一行提示词扩展成了完整的规格说明。从日志中我可以看到，生成器在应用规划和智能体设计、智能体接线以及交付 QA 前的测试等方面都做得很好。

话虽如此，QA 智能体仍然捕获了真实的缺漏。在第一轮反馈中，它指出：

> This is a strong app with excellent design fidelity, solid AI agent, and good backend. The main failure point is Feature Completeness — while the app looks impressive and the AI integration works well, several core DAW features are display-only without interactive depth: clips can't be dragged/moved on the timeline, there are no instrument UI panels (synth knobs, drum pads), and no visual effect editors (EQ curves, compressor meters). These aren't edge cases — they're the core interactions that make a DAW usable, and the spec explicitly calls for them.

在第二轮反馈中，它再次捕获了几处功能缺口：

> Remaining gaps:
> - Audio recording is still stub-only (button toggles but no mic capture)
> - Clip resize by edge drag and clip split not implemented
> - Effect visualizations are numeric sliders, not graphical (no EQ curve)

当生成器独自工作时，仍然容易遗漏细节或对功能做存根处理（stub），QA 在捕获这些最后一公里问题方面仍然发挥了价值，让生成器得以修复。

根据提示词，我期望的是一个能创建旋律、和声和鼓点节奏的程序，可以将它们编排成一首歌，并在过程中得到集成智能体的帮助。

这个应用离专业的音乐制作程序还差得很远，智能体的作曲能力显然还需要大量打磨。此外，Claude 实际上无法"听"，这使得 QA 反馈循环在音乐品味方面的效果大打折扣。

但最终的应用具备了一个可用的音乐制作程序的所有核心要素：一个能在浏览器中运行的编曲视图、混音器和走带控制。不仅如此，我还完全通过提示词拼出了一段简短的歌曲片段：智能体设置了速度和调式，铺下旋律，构建了鼓轨，调整了混音器电平，并添加了混响。歌曲创作的核心原语都已具备，智能体可以自主地驱动它们，使用工具从头到尾完成一段简单的制作。

## 未来展望

随着模型持续进步，我们大致可以预期它们能够处理更长时间的任务，也能应对更复杂的问题。在某些情况下，这意味着模型周围的脚手架（scaffolding）会随着时间推移变得不那么重要——开发者只需等待下一代模型，某些问题就会自行解决。但另一方面，模型越强大，编排框架能发挥的空间就越大，可以完成模型在基线水平上无法独立完成的复杂任务。

带着这个认识，这项工作中有几条经验值得铭记。首先，针对你正在使用的模型进行实验、在真实问题上阅读它的执行轨迹、调优它的表现以达成预期目标——这些始终是好的实践。其次，面对更复杂的任务时，有时可以通过任务分解、为问题的各个方面分配专门的智能体来获得提升空间。最后，当新模型发布时，重新审视编排框架通常是明智之举：剥离那些对性能不再承重的部分，同时加入新的组件，以实现此前不可能达到的更强能力。

从这项工作中，我的信念是：随着模型的进步，有趣的编排框架组合空间并不会缩小。相反，它在移动。AI 工程师真正有趣的工作，是不断找到下一个新颖的组合。
